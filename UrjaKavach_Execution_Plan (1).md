## 1. PRODUCT DEFINITION (LOCKED SCOPE — DO NOT RE-LITIGATE MID-BUILD)

Urja Kavach is a **live-data-grounded decision-support dashboard**, not a fully autonomous
policy system. A user (playing the role of a Ministry of Petroleum & Natural Gas control-room
analyst) opens the dashboard and can:

1. See a **live geopolitical risk score** for the Hormuz corridor and India's major alternate
   supply corridors, computed from real news-event and shipping data — not a static number.
2. See a **geospatial digital twin** of India's actual energy supply network (SPR caverns,
   major refineries, major crude-receiving ports, major pipelines), with a live AIS overlay
   showing vessel density in the Strait of Hormuz bounding box.
3. Pick a **disruption scenario** (at minimum: "Hormuz partial closure," parameterized by a
   0–100% capacity-available slider) and see the platform compute cascading effects — import
   volume impact, a simplified refinery-input-mix effect, an SPR drawdown projection — using
   real 2026 crisis data as the calibration ground truth (FINAL_ALIGNED_DOSSIER §1–2).
4. See a **ranked list of alternative sourcing recommendations** for a given scenario, benchmarked
   in the UI itself against what India's refiners actually did in 2026 (the 55%→70% non-Hormuz
   diversification, FINAL_ALIGNED_DOSSIER §2).
5. See a **plain-language risk narrative**, generated by an LLM from the underlying structured
   data (not free-floating LLM speculation) — the "context shells" pattern.

**What is explicitly out of scope, stated plainly, so it is never silently attempted:**
- No live refinery-level operational telemetry (not public — FINAL_ALIGNED_DOSSIER §3.5).
- No tanker-level cargo/contract/ownership data (proprietary — same section).
- No real government system integration of any kind. This is a standalone demo application.
- No user accounts, multi-user auth, or role management beyond a single demo login — this is a
  hackathon prototype, not a production system, and pretending otherwise wastes phases.
- No mobile app. Desktop-browser dashboard only.
- No training of a custom ML/GNN model from scratch. Risk scoring is a transparent, explainable,
  weighted-signal model (§6), not a black-box model presented as more sophisticated than it is —
  this is a direct, deliberate lesson from the AgriBloom-Agentic-ET2026 audit already in this
  project's context: an unverifiable "trained model" claim is a severe credibility risk, and a
  transparent, well-reasoned scoring formula that a judge can actually follow is a **stronger**
  submission than an opaque model with an inflated accuracy claim.

## 2. WHY THIS SCOPE, NOT A BROADER ONE (reasoning, not just a decision)

The FINAL_ALIGNED_DOSSIER's Part 4 lays out five possible sub-systems from the brief
(Geopolitical Risk Intelligence Agent, Disruption Scenario Modeller, Adaptive Procurement
Orchestrator, Strategic Reserve Optimisation Agent, Supply Chain Digital Twin). Attempting all
five at full depth in 8 days, solo or as a small team, guarantees several of them end up as
AgriBloom-style stubs — untrained exports and unreconciled numbers dressed up as finished
features. This plan instead:

- Builds the **Digital Twin and Geopolitical Risk Agent to full depth** (Tier 1) — because
  these are the two sub-systems the brief's own evaluation criteria weight most concretely
  ("geospatial evidence depth," "disruption signal detection lead time and accuracy") and
  because both have genuinely free, real, live data sources already verified
  (FINAL_ALIGNED_DOSSIER §5, and the fresh verification in §5 of this document below).
- Builds the **Scenario Modeller to one well-calibrated scenario** (Tier 1) rather than three
  shallow ones — "Hormuz partial closure" is explicitly named in the brief and has the single
  richest real 2026 ground-truth dataset of any possible scenario.
- Treats the **Procurement Orchestrator and Reserve Optimisation Agent as Tier 2** — simpler,
  rule-based, clearly-labeled decision-support calculators built from real static data
  (FINAL_ALIGNED_DOSSIER §0's verified ISPRL numbers, §2's real diversification percentages),
  not autonomous agents making independent judgment calls. If Tier 1 finishes with days to
  spare, these get built for real. If not, they are never claimed to exist.
- Explicitly never attempts a custom-trained knowledge-graph neural network (GraphSAGE etc.)
  under this timeline — the "knowledge graph" requirement is satisfied honestly with a real,
  data-backed node/edge graph and a transparent risk-propagation calculation (§6), which is a
  defensible, explainable implementation of the brief's suggested technology, not a shortcut.

## 5. RISK SCORING MODEL (Tier 1, §6 reference — kept transparent and explainable on purpose)

A weighted, explainable scoring function, computed per corridor (Hormuz vs. non-Hormuz), refreshed
on a schedule (not literally streaming):

```
risk_score(corridor, t) =
    w1 * normalized_gdelt_article_volume_zscore(corridor, window=24h vs 30d baseline)
  + w2 * normalized_price_volatility(EIA Brent/WTI daily % change, 3-day rolling)
  + w3 * ais_transit_deviation(corridor, live count vs 30-day baseline count)
  + w4 * sanctions_event_flag(OFAC SDN delta in trailing 7 days, binary/decayed)

  w1..w4 sum to 1.0; default weights: 0.35 / 0.25 / 0.30 / 0.10 — stated explicitly
  in the UI's "assumptions panel" (FINAL_ALIGNED_DOSSIER Part 6 evaluation-focus requirement),
  editable by the user in the demo to show the model is transparent, not a black box.
```

**CORRECTION (superseded from Revision 1, confirm before Phase 4 per Agent Execution Rules §2B):**
the original 5-term formula above included a `gdelt_goldstein_severity` term. GDELT's DOC 2.0
API — the one this project actually integrates against (Architecture Plan §2) — does not expose
a per-article Goldstein score directly; that lives in the separate, heavier GDELT Events table.
The corrected, 4-term formula above (article-volume z-score standing in as the GDELT signal,
weights rebalanced to sum to 1.0) is the authoritative version, matching HLD/LLD §2.5's LLD.
This is a deliberate, documented correction, not silent scope drift — **it still needs an
explicit go-ahead from the team before Phase 4 begins.**

This is a **judgment call, not a discovered constant** — say so explicitly in the demo. The
credibility claim is "every input to this score is real and traceable," not "these exact
weights are empirically optimal." That distinction matters and should be stated out loud in
the presentation, not glossed over.

## 6. DATABASE SCHEMA (LOCKED — Postgres, via SQLAlchemy/Prisma equivalent per Architecture Plan)

```sql
-- Static seed data, loaded once at boot from data/india_energy_nodes.json
CREATE TABLE nodes (
  id            TEXT PRIMARY KEY,          -- e.g. 'spr_visakhapatnam', 'refinery_jamnagar'
  node_type     TEXT NOT NULL,             -- 'spr' | 'refinery' | 'port' | 'pipeline_segment'
  name          TEXT NOT NULL,
  lat           DOUBLE PRECISION NOT NULL,
  lon           DOUBLE PRECISION NOT NULL,
  capacity_value DOUBLE PRECISION,         -- MMT for spr/refinery, mb/d for pipeline
  capacity_unit TEXT,
  source_note   TEXT NOT NULL              -- required: where this number came from (dossier §)
);

CREATE TABLE edges (
  id            TEXT PRIMARY KEY,
  from_node_id  TEXT NOT NULL REFERENCES nodes(id),
  to_node_id    TEXT NOT NULL REFERENCES nodes(id),
  edge_type     TEXT NOT NULL,             -- 'pipeline' | 'shipping_corridor'
  capacity_value DOUBLE PRECISION,
  source_note   TEXT NOT NULL
);

CREATE TABLE risk_scores (
  id            SERIAL PRIMARY KEY,
  corridor      TEXT NOT NULL,             -- 'hormuz' | 'non_hormuz_west_africa' | 'non_hormuz_americas' | 'non_hormuz_russia'
  computed_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  score         DOUBLE PRECISION NOT NULL, -- 0-100
  component_gdelt_volume     DOUBLE PRECISION,  -- corrected: article-volume z-score, not Goldstein severity (see §5)
  component_price_volatility DOUBLE PRECISION,
  component_ais_deviation    DOUBLE PRECISION,
  component_sanctions_flag   DOUBLE PRECISION,
  weights_used  JSONB NOT NULL             -- the exact w1..w5 used, for auditability
);

CREATE TABLE ais_snapshots (
  id            SERIAL PRIMARY KEY,
  captured_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  bounding_box  TEXT NOT NULL,             -- 'hormuz' | 'jamnagar_vadinar'
  vessel_count  INTEGER NOT NULL,
  raw_payload_path TEXT                    -- path to a stored JSON snippet, for the demo's own auditability
);

CREATE TABLE scenarios (
  id            TEXT PRIMARY KEY,          -- 'hormuz_partial_closure'
  name          TEXT NOT NULL,
  description   TEXT NOT NULL,
  ground_truth_source TEXT NOT NULL        -- cites FINAL_ALIGNED_DOSSIER section
);

CREATE TABLE scenario_runs (
  id            SERIAL PRIMARY KEY,
  scenario_id   TEXT NOT NULL REFERENCES scenarios(id),
  capacity_available_pct DOUBLE PRECISION NOT NULL,  -- the slider value, 0-100
  run_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  projected_import_volume_change_pct DOUBLE PRECISION,
  projected_spr_days_cover DOUBLE PRECISION,
  narrative_text TEXT                      -- LLM-generated, stored for reproducibility/demo replay
);
```

Indexes: `risk_scores(corridor, computed_at)`, `ais_snapshots(bounding_box, captured_at)`,
`scenario_runs(scenario_id, run_at)`.

## WHAT'S STILL GENUINELY OPEN (stated honestly)

- AISstream.io's documented "subscription accepted, no data delivered" issue (aisstream/aisstream#15)
  is unresolved as of this document's writing — Phase 3's done-condition and this plan's golden

  this project yet — build the ingestion scheduler conservatively (poll every 10–15 minutes, not
  every few seconds) from the start rather than discovering the limit the hard way mid-build.
- The risk-scoring weights in §5 are a reasoned starting point, not a validated model — say this
  explicitly in the demo; it is more credible than implying otherwise.
